## 2. 문제 정의 및 배경

### 2.1 문제 정의
- 고객은 해외 디자이너 상품 구매 시 국가별 세금/관세/배송비를 결제 직전에야 확인해 이탈이 발생한다.
- 파트너/운영자는 국가별 가격 계산과 주문-정산 처리를 수작업으로 수행해 운영 비용과 오류가 증가한다.
- 핵심 병목은 가격 계산 일관성 부족, 주문 상태 전이 지연, 정산 집계 수동 처리다.

### 2.2 비즈니스 배경
- 정확한 예상 총액 제공은 결제 전 이탈을 감소시키고 전환율을 개선한다.
- 가격 계산 및 정산 자동화는 운영 시간을 절감하고 재작업 비용을 줄인다.
- KPI는 주간/월간으로 측정하며 4주 파일럿 데이터를 기준으로 유지/개선 여부를 판단한다.

## 4. 사용자 및 시나리오

### 4.1 사용자 유형
- 고객(Customer)
- 브랜드 파트너(Brand Partner)
- 운영자(Admin Operator)

### 4.2 핵심 시나리오
- UC-001: 고객이 배송 국가 기준 최종 예상 금액을 확인하고 결제한다.
- UC-002: 운영자가 상품을 등록하고 가격 정책을 적용한다.
- UC-003: 주문/환불 데이터를 기반으로 정산 초안이 자동 생성된다.

## 5. 범위 정의

### 5.1 포함 범위 (In Scope)

#### P0 (필수)
- 고객 웹: 상품 탐색/상세/장바구니/체크아웃/주문 조회
- 가격 엔진: 국가별 가격 견적(`PriceQuote`) 계산
- 관리자: 상품/가격정책/주문/정산 기본 운영
- AI 자동화: 정산 초안 집계 자동 생성

#### P1 (중요)
- 가격 정책 시뮬레이션 고도화
- 운영 예외 태스크 자동 생성
- 가격 산출 근거(Explainability) UI 강화

### 5.2 스프린트 스코프 잠금 (확정)

- 이번 스프린트 개발 범위는 `P0 only`로 확정한다.
- `P1` 항목은 구현 범위에서 제외하고 백로그로 관리한다.
- P1 관리 문서: `docs/backlog-p1.md`

### 5.3 제외 범위 (Out of Scope)
- P2 이상 기능(고급 개인화 추천, 실시간 경쟁가 자동 반영, 고급 리포팅 대시보드)

## 6. 기능 요구사항

| ID | Priority | 요구사항 | 수용 기준 |
|---|---|---|---|
| FR-001 | P0 | 고객은 상품 상세에서 배송 국가별 예상 총액을 확인할 수 있어야 한다. | 국가 변경 시 1초 내 `PriceQuote` 응답 표시 |
| FR-002 | P0 | 운영자는 관리자에서 상품/가격/주문 상태를 관리할 수 있어야 한다. | 권한 사용자만 CRUD 및 상태 전이 가능 |
| FR-003 | P0 | 정산 데이터는 주문/환불/수수료 기준으로 집계되어야 한다. | 정산 기간별 `Settlement` 합계 일치 |
| FR-004 | P1 | 가격 계산 결과는 적용 룰 설명을 제공해야 한다. | `explain` 배열에 최소 1개 근거 포함 |
| FR-005 | P0 | 체크아웃은 금액 불일치 시 결제를 차단해야 한다. | `expectedFinalPrice`와 서버 계산값 불일치 시 `409` 반환 |

## 7. 비기능 요구사항

### 7.1 성능
- 핵심 API p95 응답시간:
  - `POST /pricing/quote`: 700ms 이하
  - `GET /products`: 900ms 이하
  - `GET /products/:productId`: 700ms 이하
  - `POST /checkout/confirm`: 1200ms 이하
- 프론트 성능:
  - 상품 목록/상세 LCP: 2.8s 이하(모바일 4G 기준)
