# pin-stitch 제품 요구사항

## 1. 제품 정의

`pin-stitch`는 체형 정보와 구매 리뷰를 연결해 사용자가 자신에게 맞는 패션 상품을 더 쉽게 고르도록 돕는 커머스 플랫폼입니다.

핵심 가치는 다음과 같습니다.

> 나와 비슷한 체형의 리뷰로 더 정확하게 고르고, 셀러는 체형별 데이터를 기반으로 상품을 개선한다.

## 2. 문제 정의

### 고객 문제

- 모델 착용 이미지와 실제 착용감 사이의 차이가 큽니다.
- 같은 상품도 키, 골격 타입, 평소 사이즈, 선호 핏에 따라 만족도가 달라집니다.
- 리뷰가 많아도 나와 비슷한 체형의 후기를 찾기 어렵습니다.
- 사이즈표만으로 실제 핏, 기장, 어깨, 허리 여유를 예측하기 어렵습니다.
- 온라인에서 본 상품을 오프라인 매장에서 입어보고 싶어도 가까운 매장 재고를 확인하기 어렵습니다.

### 셀러 문제

- 리뷰와 반품 사유가 쌓여도 어떤 체형의 고객이 왜 불만족했는지 파악하기 어렵습니다.
- 사이즈/핏 불만이 상세페이지, 옵션 구성, 상품 기획에 충분히 반영되지 않습니다.
- 반복되는 핏 불만족은 반품 비용과 고객 이탈로 이어집니다.

## 3. 사용자

| 사용자 | 설명 |
| --- | --- |
| Customer | 패션 상품을 탐색하고 주문하는 일반 사용자 |
| Seller | 입점 상품의 리뷰/반품 데이터를 확인하고 상품을 개선하는 브랜드/셀러 |
| Admin | 샘플 데이터와 운영 상태를 관리하는 플랫폼 운영자. MVP에서는 별도 화면을 만들지 않습니다. |

## 4. MVP 목표

- 사용자가 본인과 유사한 체형의 리뷰를 빠르게 확인할 수 있게 합니다.
- 상품별 체형 적합도와 사이즈 주의사항을 제공합니다.
- 샘플 매장 재고를 보여줘 온라인 탐색과 오프라인 피팅을 연결합니다.
- 장바구니와 주문 생성까지 이어지는 기본 커머스 흐름을 완성합니다.
- 셀러가 상품별 체형 만족도와 핏 불만 원인을 파악할 수 있게 합니다.

## 5. 비목표

- 실제 결제 PG 연동
- 실제 배송/운송장 연동
- 실제 POS/ERP 재고 연동
- 실제 인증 서버 구축
- 사진 기반 체형 분석
- AI Virtual Try-on
- LLM 기반 리뷰 요약 자동화
- 지오펜스/비콘 기반 매장 모드
- 스마트 피팅룸 요청
- 라이브 커머스와 실시간 채팅

## 6. 핵심 시나리오

### 고객 흐름

1. 사용자가 체형 프로필을 등록합니다.
2. 상품 목록에서 검색, 필터, 정렬로 상품을 탐색합니다.
3. 상품 카드에서 체형 적합도 배지를 확인합니다.
4. 상품 상세에서 유사 체형 리뷰 요약과 사이즈/핏 주의사항을 확인합니다.
5. 리뷰 목록에서 키, 골격 타입, 구매 사이즈, 핏 평가 기준으로 리뷰를 필터링합니다.
6. 원하는 색상/사이즈 옵션을 선택합니다.
7. 매장 재고 섹션에서 가까운 매장의 보유 여부와 랙 위치를 확인합니다.
8. 장바구니에서 최대 할인 쿠폰 조합이 자동 적용된 금액을 확인합니다.
9. 배송지를 입력하고 주문을 생성합니다.
10. 주문 완료와 주문 내역에서 주문 상태와 할인 내역을 확인합니다.

### 셀러 흐름

1. 셀러가 대시보드에 진입합니다.
2. 전체 상품 수, 평균 평점, 핏 불만 비율, 반품률을 확인합니다.
3. 문제 상품 목록에서 이슈가 큰 상품을 선택합니다.
4. 상품별 체형 만족도와 사이즈별 핏 평가 분포를 확인합니다.
5. 긍정/부정 리뷰 키워드를 확인합니다.
6. 상품 개선 인사이트 카드로 다음 액션을 결정합니다.

## 7. 기능 요구사항

| ID | Priority | 요구사항 | 수용 기준 |
| --- | --- | --- | --- |
| FR-001 | P0 | 체형 프로필 등록/수정 | 키, 몸무게, 상의/하의 사이즈, 골격 타입, 선호 핏, 체형 정보 활용 동의를 저장할 수 있습니다. |
| FR-002 | P0 | 상품 목록 탐색 | 검색어, 카테고리, 브랜드, 가격, 체형 적합도 기준으로 탐색할 수 있습니다. |
| FR-003 | P0 | 상품 상세 | 이미지, 브랜드, 상품명, 가격, 소재, 사이즈표, 옵션, 리뷰 요약, 적합도, 재고, 장바구니 CTA를 제공합니다. |
| FR-004 | P0 | 체형 적합도 | 로그인 또는 샘플 사용자의 `BodyProfile` 기준 0~100점 점수와 신뢰도를 표시합니다. |
| FR-005 | P0 | 체형 기반 리뷰 요약 | 유사 체형 리뷰 기반 요약을 제공하고, 유사 리뷰가 없으면 전체 리뷰 기반으로 fallback합니다. |
| FR-006 | P0 | 리뷰 필터 | 키 범위, 구매 사이즈, 골격 타입, 핏 평가, 키워드로 리뷰를 필터링합니다. |
| FR-007 | P0 | 매장 재고 | 선택 옵션 기준 매장명, 거리, 영업 상태, 보유 수량, 층/존/랙 위치를 표시합니다. |
| FR-008 | P0 | 장바구니 | 상품/옵션/수량, 수량 변경, 삭제, 상품 금액 합계, 할인 내역, 최종 금액을 표시합니다. |
| FR-009 | P0 | 쿠폰 자동 적용 | 보유 쿠폰 중 최종 주문 예정 금액이 가장 낮은 조합을 자동 선택합니다. |
| FR-010 | P0 | 주문 생성 | 실제 결제 없이 장바구니 스냅샷과 배송지로 주문을 생성합니다. |
| FR-011 | P0 | 셀러 대시보드 | 핵심 지표, 문제 상품 Top 5, 최근 인사이트를 표시합니다. |
| FR-012 | P0 | 셀러 상품 분석 | 체형별 평균 평점, 사이즈별 핏 분포, 키워드, 대표 리뷰, 개선 인사이트를 표시합니다. |
| FR-013 | P0 | 셀러 리뷰/반품 분석 | 리뷰 필터, 키워드 요약, 반품 사유 비율, 상품별 반품률, 체형별 반복 이슈를 표시합니다. |

## 8. P1/P2 범위

### P1

- 리뷰 작성
- 반품 사유 입력/분류
- 체형별 추천 상품 영역
- 셀러 리포트 다운로드
- 상품 개선 가이드 자동 생성
- 쿠폰 자동 적용 정책 고도화
- 실제 주문/결제 연동
- 실제 POS/ERP 기반 재고 연동

### P2

- 사진 기반 체형 분석
- AI Virtual Try-on
- 스마트 피팅룸 요청
- 개인 스타일링 챗봇
- 체형 기반 추천 광고 상품

### P3

- 라이브 커머스 방송
- 방송 중 실시간 채팅
- 방송 중 현재 소개 상품 전환
- 실시간 좋아요/시청자 수 표시
- 방송 중 장바구니 담기
- 채팅 금칙어, 신고, 관리자 삭제
- 방송 종료 후 시청자 수, 채팅 수, 상품 클릭 수, 장바구니 전환 수, 주문 수 요약

## 9. 라이브 커머스 확장 방향

라이브 커머스는 MVP 이후 별도 확장 기능으로 다룹니다. 목표는 영상 인프라 자체를 직접 구현하는 것이 아니라, 쇼핑몰 도메인과 실시간 상호작용을 연결하는 것입니다.

### 기능 범위

| 영역 | 구현 방향 |
| --- | --- |
| 방송 관리 | 셀러가 방송을 생성하고 방송에 소개할 상품을 등록합니다. |
| 영상 송출 | AWS IVS, Mux, Agora, LiveKit, Cloudflare Stream 같은 외부 라이브 스트리밍 서비스를 사용합니다. |
| 실시간 채팅 | NestJS WebSocket Gateway로 채팅 송수신, 저장, 관리자 삭제를 처리합니다. |
| 상품 전환 | 셀러가 방송 중 현재 소개 상품을 전환하면 시청자 화면에 실시간 반영합니다. |
| 구매 연결 | 시청자는 방송 화면에서 상품 상세를 열거나 장바구니에 상품을 담을 수 있습니다. |
| 운영 관리 | 금칙어 필터, 신고, 관리자 삭제, 사용자 차단을 제공합니다. |
| 방송 리포트 | 방송 종료 후 시청자 수, 채팅 수, 좋아요 수, 상품 클릭 수, 장바구니 전환 수, 주문 수를 요약합니다. |

### 데이터 저장 원칙

정합성이 중요한 데이터는 PostgreSQL에 저장합니다.

- 방송 방
- 방송 상품
- 상품
- 주문
- 결제
- 재고
- 쿠폰
- 사용자

대량으로 쌓이고 구조가 유연한 실시간 데이터는 별도 저장소 도입을 검토합니다.

- 채팅 메시지
- 좋아요 이벤트
- 시청 로그
- 상품 클릭 로그
- 채팅 신고/관리 로그

초기에는 PostgreSQL로 단순하게 시작할 수 있고, 데이터량과 실시간 처리 요구가 커지면 MongoDB, Redis, OpenSearch 같은 보조 저장소를 도입합니다.

## 11. 리스크와 대응

| Risk | 영향도 | 대응 |
| --- | --- | --- |
| 초기 리뷰 데이터 부족 | High | 샘플 데이터를 충분히 만들고, 리뷰 작성은 P1로 확장합니다. |
| 체형 정보 입력 저항 | Medium | 필수 입력을 최소화하고 선택 입력을 분리합니다. |
| 리뷰 요약 신뢰도 부족 | Medium | 요약에 사용된 리뷰 수, 체형 조건, 대표 근거 리뷰를 함께 보여주고 내부 추적용 리뷰 ID는 화면에 노출하지 않습니다. |
| 재고 데이터 현실성 부족 | Medium | MVP에서는 샘플 데이터임을 전제로 하고 실제 연동은 P1로 분리합니다. |
| 개인정보 우려 | High | 체형 정보 활용 동의를 받고 셀러 화면에서는 리뷰어 정보를 비식별 처리합니다. |
| 라이브 커머스 영상 인프라 복잡도 | High | 직접 영상 서버를 구현하지 않고 외부 라이브 스트리밍 서비스를 사용합니다. |
| 실시간 채팅 운영 부담 | Medium | 금칙어, 신고, 관리자 삭제, 사용자 차단을 확장 범위에 포함합니다. |

## 12. 결정 사항

- MVP 화면은 `apps/web` 하나에서 고객 화면과 `/seller` 화면을 함께 구현합니다.
- MVP API는 `apps/api` NestJS에서 구현하고, PostgreSQL seed 데이터로 데모 가능한 상태를 만듭니다.
- 리뷰 요약, 체형 적합도, 쿠폰 적용, 셀러 인사이트는 규칙 기반으로 시작합니다.
- 주문은 실제 결제 없이 `ORDER_CREATED` 상태의 주문 생성 흐름으로 구현합니다.
- 라이브 커머스는 MVP 이후 P3 확장 기능으로 분리하고, 영상 송출은 외부 서비스를 사용합니다.
- 리뷰 요약은 내부적으로 `basisReviewIds`로 근거 리뷰를 추적하지만, 사용자 화면에는 리뷰 ID가 아니라 리뷰 수, 체형 조건, 대표 근거 리뷰를 표시합니다.
- 내부 ID, 원본 timestamp, 좌표, 디버깅용 request ID 등은 일반 사용자 화면에 직접 노출하지 않고 사용자 친화적인 번호, 상태, 날짜, 거리, 문구로 가공해 표시합니다.
